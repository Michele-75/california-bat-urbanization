GBIF Notes

GBIF occurrence download

-Occurrence records were requested from GBIF using the Occurrence Download API via rgbif::occ_download().

-Focal taxa were resolved against the GBIF Backbone Taxonomy using rgbif::name_backbone().

-The resulting backbone resolution table (species names → GBIF keys) is saved as data/raw/gbif/gbif_backbone_taxa.csv to document the taxonomic concepts used in the query.

-Download filters:

*taxonKey in focal taxa (backbone-resolved)

*hasCoordinate == TRUE

*year >= 2012

-The GBIF download key and record count are recorded in data/raw/gbif/gbif_download_meta.csv.

Taxonomy handling and synonymy

-GBIF provides both interpreted taxon fields (e.g., species) and original/display name fields (e.g., scientific_name).

-Taxonomic synonymy and genus reclassification can cause the interpreted species string to differ from the queried name (e.g., hoary bat occurrences may be interpreted as Aeorestes cinereus while retaining Lasiurus cinereus in scientific_name).

-To prevent accidental data loss due to name-string mismatches, downstream filtering and aggregation use GBIF backbone identifiers (species_key) as the canonical species identifier.

-Human-readable species names used in figures/tables are applied via a mapping from species_key to a project-standard label.

Processed outputs

-data/processed/gbif/gbif_bats_ca_clean.gpkg and .rds: cleaned point occurrences within California counties (geometry preserved).

-data/processed/gbif/bats_county_year_species.csv and .rds: county × year × species panel (no geometry), suitable for modeling and merging with VIIRS county-year summaries.

-QA summaries are written alongside outputs to document record attrition and final distributions
